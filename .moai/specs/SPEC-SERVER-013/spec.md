# SPEC-SERVER-013 — 대시보드 + 통계

> status: DOING · deps: 005, 006, 007 · Phase 1 (M2 도메인) · **마지막 SPEC**

## 목표

대시보드/통계 REST API. 오늘/월 매출·지출 요약, 다가오는 예약, 발동 리마인더, 최근 매출,
카테고리/결제수단/채널/고객/지출 통계를 네이티브 SQL로 집계한다. 완료 시 Phase 1 전부 DONE.

## 범위 (In)

- **오늘 대시보드** `GET /dashboard/today`: 오늘 매출 요약(미수 제외) + 다가오는 예약 + 발동 리마인더 + 최근 매출 5건 + 매출 카테고리.
- **월 통계** `GET /dashboard/month?month=`: 월 매출 요약 + 지출 총액 + 카테고리/결제수단/채널/지출 카테고리별 통계(건수·금액·비율) + 고객 통계(총/재방문/신규).
- **네이티브 SQL 집계**: Postgres `FILTER`·`GROUP BY`·`DISTINCT`·`EXISTS`. 모든 쿼리 `user_id` 바인딩(테넌트 격리·SQL 인젝션 방지). 미수(unpaid)는 매출 합계·통계에서 제외.
- 예약/리마인더/최근매출/카테고리는 기존 서비스(ReservationService·SaleService·SaleCategorySettingService) 재사용.

## 범위 밖 (Out)

- 차트 렌더링/기간 비교(전월 대비 등) — 앱 표시 영역.
- 트렌드 카운트 위젯(인사이트) — SPEC-011에서 별도 제공 가능.

## 인수 기준

1. `./gradlew build test` 통과.
2. 오늘 요약: 미수 제외 총매출, 결제수단별 금액, 입금대기 건수/금액. 부속(예약·리마인더·최근매출·카테고리) 포함.
3. 월 통계: 매출/지출 요약, 카테고리/결제수단/채널/지출 통계(금액·비율), 고객 통계(총/재방문/신규).
4. 비율 합은 100에 수렴(반올림 오차 허용).
5. 모든 집계는 네이티브 SQL + user_id 바인딩.
6. **멀티테넌시 격리**: 다른 user의 집계는 0.

## 설계 메모

- 집계는 JdbcTemplate 네이티브 SQL(DESIGN의 "통계는 네이티브 SQL" 방침). 합계/통계는 `payment_method <> 'unpaid'`로 미수 제외.
- 고객 재방문 = 월 내 전화번호 중 월 시작 이전 매출이 존재(EXISTS)하는 distinct 수. 신규 = 총 - 재방문.
- KST 기준 오늘/이번 달. 카테고리/결제/채널 라벨은 알려진 값 매핑 + 폴백.
- 검증은 Zonky 임베디드 PG(시드 데이터로 합계·비율·격리 검증).
