# Flori Server — 데이터베이스 명세

Flori 백엔드의 PostgreSQL 스키마 전체 명세. 각 테이블의 역할·컬럼·제약·인덱스·관계를 정리한다.

- **SSOT**: `src/main/resources/db/migration/*.sql` (Flyway). 본 문서는 그 스키마를 사람이 읽기 쉽게 해설한 것이며, 코드 변경 시 함께 갱신한다.
- **DB**: PostgreSQL 16 (로컬: Docker `flori-pg`, 운영: AWS RDS)
- **PK 전략**: 모든 테이블 `id BIGINT GENERATED BY DEFAULT AS IDENTITY` (시퀀스 기반). FK도 BIGINT로 정렬. (구 UUID 전략에서 전환 — 인덱스 크기·조인 비용 절감)
- **시각 타입**: 모든 타임스탬프는 `TIMESTAMPTZ`. `updated_at`은 `update_updated_at()` 트리거로 자동 갱신.
- **멀티테넌시** [HARD]: 공유 테이블을 제외한 모든 도메인 테이블은 `user_id BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE`를 가지며, 애플리케이션이 JWT의 `userId`로 격리한다. RLS 없음 — 앱이 유일한 방어선.

| 버전 | 설명 | 코드(repo) | 로컬 DB(`flori-pg`) |
|------|------|:---:|:---:|
| V1 | init schema | ✅ | ✅ |
| V2 | seed instagram accounts | ✅ | ✅ |
| V3 | onboarding | ✅ | ✅ |
| V4 | social only | ✅ | ✅ |
| V5 | drop onboarded | ✅ | ✅ |

**V5(`V5__drop_onboarded.sql`)까지 모두 적용 완료.** `users` 테이블은 `password_hash` 컬럼이 제거되고 `email`이 NOT NULL로 강화된 **소셜 전용 인증** 상태이며, 더 이상 의미가 없어진 `onboarded` 컬럼(V3에서 추가)이 V5에서 제거됐다. **User 행 존재 자체가 온보딩 완료를 의미**한다.

데이터 현황(로컬): `instagram_accounts` 15건(V2 시드), 그 외 도메인 테이블은 비어 있음.

## ER 관계 요약

```
users (1) ──┬── (N) refresh_tokens          [CASCADE]
            ├── (1) user_profiles            [PK=FK, CASCADE]
            ├── (1) user_preferences         [PK=FK, CASCADE]
            ├── (N) customers                [CASCADE]
            ├── (N) sales ──────┐            [CASCADE]
            ├── (N) reservations ┘ (양방향 FK: reservations.sale_id ↔ sales)
            ├── (N) expenses ── (N:1) recurring_expenses [recurring SET NULL]
            ├── (N) recurring_expenses ── (N) recurring_skips [CASCADE]
            ├── (N) photo_cards ── (N:1) sales [SET NULL]
            ├── (N) calendar_events / sale_categories / payment_methods /
            │       expense_categories / expense_payment_methods / photo_tags
            ├── (N) insight_scraps  (polymorphic → trend_articles | instagram_posts)
            ├── (N) push_subscriptions / notification_log
            └── (1) subscriptions / (N) subscription_events

instagram_accounts (1) ── (N) instagram_posts   [공유, CASCADE]
customers (1) ── (N) sales                       [customer_id SET NULL]
```

## 1. 인증·사용자

### `users` — 사용자 (자체 인증, 소셜 전용)

원본 Supabase 인증을 대체하는 자체 사용자 테이블. 모든 도메인 테이블의 `user_id`가 참조한다. 카카오/구글/네이버 소셜 로그인 + 온보딩 완료 시점에 생성.

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | BIGINT | PK, IDENTITY | |
| `email` | VARCHAR(255) | NOT NULL, UNIQUE | 온보딩에서 항상 채워짐 (소셜 전용) |
| `name` | VARCHAR(100) | | 계정 표시명/소셜 닉네임 (가게명과 분리, 가게명은 `user_profiles.store_name`) |
| `provider` | VARCHAR(20) | NOT NULL, DEFAULT 'LOCAL' | 소셜 제공자 (KAKAO/GOOGLE/NAVER 등) |
| `provider_id` | VARCHAR(255) | | 소셜 측 사용자 식별자 |
| `is_active` | BOOLEAN | NOT NULL, DEFAULT TRUE | |
| `created_at` / `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 트리거로 updated_at 자동 갱신 |

> V3에서 추가했던 `onboarded` 컬럼은 V5(`V5__drop_onboarded.sql`)에서 제거됐다. User 행은 `register/complete`(온보딩 완료) 시점에만 `user_profiles`와 함께 원자적으로 생성되므로 **User 존재 = 온보딩 완료**이고, 별도 플래그가 불필요하다.

- **인덱스**: PK(id), UNIQUE(email), `uq_users_provider_identity` UNIQUE(provider, provider_id) WHERE provider_id IS NOT NULL — 같은 소셜 신원=같은 사용자
- **트리거**: `update_users_updated_at`

### `refresh_tokens` — 리프레시 토큰 저장소

JWT refresh 회전/무효화 추적. **원문이 아닌 SHA-256 해시만 저장**. 회전 시 이전 토큰 revoke, 로그아웃 시 revoke.

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | BIGINT | PK, IDENTITY | |
| `user_id` | BIGINT | NOT NULL, FK→users CASCADE | |
| `token_hash` | VARCHAR(64) | NOT NULL, UNIQUE | SHA-256 해시 (64 hex) |
| `expires_at` | TIMESTAMPTZ | NOT NULL | 만료 시각 |
| `revoked` | BOOLEAN | NOT NULL, DEFAULT FALSE | 무효화 여부 |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | |

- **인덱스**: `idx_refresh_tokens_user_id`, `idx_refresh_tokens_expires_at`

### `user_profiles` — 사용자 프로필 (가게 정보, users와 1:1)

온보딩에서 수집하는 가게 프로필. PK가 곧 FK(`user_id`)이므로 본질적으로 테넌트 격리(임의 user_id 주입 불가).

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `user_id` | BIGINT | **PK이자 FK**→users CASCADE | |
| `store_name` | TEXT | NOT NULL | 가게명 (`users.name`과 분리) |
| `region_sido` | TEXT | NOT NULL | 시/도 |
| `region_sigungu` | TEXT | | 시/군/구 |
| `owner_age_range` | TEXT | | 사장 연령대 |
| `interests` | TEXT[] | NOT NULL, DEFAULT '{}' | 관심사 (배열) |
| `specialties` | TEXT[] | NOT NULL, DEFAULT '{}' | 전문 분야 (배열) |
| `created_at` / `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 트리거 갱신 |

### `user_preferences` — 사용자 설정 (하단바 커스터마이즈, users와 1:1)

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `user_id` | BIGINT | **PK이자 FK**→users CASCADE | |
| `bottom_nav_items` | JSONB | NOT NULL, DEFAULT `["calendar","sales","expenses","customers","insights"]` | 하단 네비 항목 순서 |
| `updated_at` | TIMESTAMPTZ | DEFAULT NOW() | 트리거 갱신 |

## 4. 지출

### `expenses` — 지출

`total_amount`는 서버가 `unit_price * quantity`로 계산(SSOT). 카테고리 CHECK 없음 — 커스텀 카테고리 허용.

| `user_id` | BIGINT | NOT NULL, FK→users CASCADE | |
| `date` | DATE | NOT NULL | |
| `item_name` | VARCHAR(200) | NOT NULL | |
| `category` | VARCHAR(30) | NOT NULL | 커스텀 허용 |
| `unit_price` | INTEGER | NOT NULL | 단가 |
| `quantity` | INTEGER | DEFAULT 1 | 수량 |
| `total_amount` | INTEGER | NOT NULL | 서버 계산값 |
| `payment_method` | VARCHAR(20) | NOT NULL, CHECK(cash/card/transfer/naverpay/kakaopay) | |
| `card_company` | VARCHAR(50) | | |
| `vendor` | VARCHAR(100) | | 거래처 |
| `note` | TEXT | | |
| `recurring_id` | BIGINT | FK→recurring_expenses SET NULL | 고정비 템플릿(자동생성 시) |

| `created_at` / `updated_at` | TIMESTAMPTZ | DEFAULT NOW() | 트리거 갱신 |

- **제약**: UNIQUE(recurring_id, date) — 같은 템플릿·같은 날짜 중복 자동생성 방지(NULL은 distinct 취급)

## 부록: 공통 규칙

- **`update_updated_at()` 트리거 함수**: `updated_at` 컬럼이 있는 거의 모든 테이블에 BEFORE UPDATE 트리거로 부착. 예외: `subscriptions`/`subscription_events`(앱 로직이 직접 갱신), `notification_log`/`recurring_skips`(updated_at 없음).
- **CASCADE 삭제**: 사용자 삭제 시 모든 사용자별 데이터가 연쇄 삭제(`ON DELETE CASCADE`). 단 `sales.customer_id`, `reservations.sale_id`, `photo_cards.sale_id`, `expenses.recurring_id`는 `SET NULL`(참조 대상 삭제 시 연결만 끊음).
- **JSONB / 배열**: JPA에서 hypersistence-utils로 매핑. 통계 집계는 네이티브 SQL.
- **로컬 DB 초기화**: `docker compose down -v && docker compose up -d` → 앱 부팅 시 Flyway 자동 재적용.
