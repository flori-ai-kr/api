# Flori API — 데이터베이스 명세

Flori 백엔드의 PostgreSQL 스키마 전체 명세. 각 테이블의 역할·컬럼·제약·인덱스·관계를 정리한다.

- **SSOT**: `docs/sql/all-tables-ddl.sql` (DDL 직접 관리 — Flyway 미사용). 본 문서는 그 스키마를 사람이 읽기 쉽게 해설한 것이며, 코드 변경 시 함께 갱신한다.
- **DB**: PostgreSQL 16 (로컬: Docker `flori-pg`, 운영: AWS RDS)
- **PK 전략**: 모든 테이블 `id BIGINT GENERATED BY DEFAULT AS IDENTITY` (시퀀스 기반). FK도 BIGINT로 정렬. (구 UUID 전략에서 전환 — 인덱스 크기·조인 비용 절감)
- **시각 타입**: 모든 타임스탬프는 `TIMESTAMPTZ`. `updated_at`은 `update_updated_at()` 트리거로 자동 갱신.
- **멀티테넌시** [HARD]: 공유 테이블을 제외한 모든 도메인 테이블은 `user_id BIGINT NOT NULL`을 가지며, 애플리케이션이 JWT의 `userId`로 격리한다. RLS 없음 — 앱이 유일한 방어선.
- **간접참조(FK 제약 미사용)**: 모든 참조 컬럼(`user_id`, `sale_id`, `recurring_id`, `account_id` 등)은 **FK 제약 없이 컬럼만 유지**한다. 참조 무결성·연쇄삭제는 DB가 아닌 **애플리케이션이 책임**진다(부모 삭제 시 서비스가 자식 행을 명시적으로 정리). 마이그레이션: `docs/sql/migration/26-05-29-drop-foreign-keys.sql`.

---

## 스키마 관리 방식 (2026-05-28 기준)

Flyway 같은 런타임 마이그레이션 도구를 쓰지 않고 **DDL을 직접 관리**한다. 스키마 정본은 `docs/sql/all-tables-ddl.sql`(전체 스냅샷) + `docs/sql/seed.sql`(공유 시드)이며, 운영(RDS)·로컬(`flori-pg`)에는 이 DDL을 수동 적용한다. 애플리케이션은 부팅 시 `ddl-auto: validate`로 엔티티↔DB 정합성만 검증한다(스키마 생성/변경 안 함). 테스트는 임베디드 PostgreSQL에 `spring.sql.init`로 이 DDL을 적용한다. 자세한 운영 절차는 `docs/sql/README.md` 참조.

**현재 스키마 상태**: `users`는 **소셜 전용 인증**(비밀번호 컬럼 없음, `email` NOT NULL)이며 **User 행 존재 자체가 온보딩 완료를 의미**한다. 닉네임 컬럼 `nickname`은 NOT NULL + 전역 UNIQUE(`uq_users_nickname`)다. 가게/사장 프로필은 `user_profiles`(1:1)에 분리 저장한다.

데이터 현황(로컬): `instagram_accounts` 15건(공유 시드), 그 외 도메인 테이블은 비어 있음.

## ER 관계 요약

> FK 제약 없음(간접참조): 아래 관계선은 논리적 참조를 표현하며, DB에 FOREIGN KEY 제약은 없다.
> 부모 삭제 시 연쇄 정리는 애플리케이션(서비스)이 명시적으로 수행한다.

```
users (1) ──┬── (N) refresh_tokens          [앱이 CASCADE 처리]
            ├── (1) user_profiles            [PK=user_id, 앱이 CASCADE 처리]
            ├── (1) user_preferences         [PK=user_id, 앱이 CASCADE 처리]
            ├── (N) customers                [앱이 CASCADE 처리]
            ├── (N) sales ──────┐            [앱이 CASCADE 처리]
            ├── (N) reservations ┘ (reservations.sale_id 논리참조 → sales)
            ├── (N) expenses ── (N:1) recurring_expenses [recurring_id NULL 처리]
            ├── (N) recurring_expenses ── (N) recurring_skips [앱이 CASCADE 처리]
            ├── (N) photo_cards ── (N:1) sales [sale_id NULL 처리]
            ├── (N) schedules / sale_categories / payment_methods /
            │       expense_categories / expense_payment_methods / photo_tags
            ├── (N) insight_scraps  (polymorphic → trend_articles | instagram_posts)
            ├── (N) push_subscriptions / notification_log
            └── (1) subscriptions / (N) subscription_events

instagram_accounts (1) ── (N) instagram_posts   [공유, 앱이 CASCADE 처리]
customers (1) ── (N) sales                       [customer_id NULL 처리]

community_posts (1) ──┬── (N) community_comments  [공유, 앱이 CASCADE 처리; parent_id 셀프참조 대댓글]
                      └── (N) community_likes     [공유, 앱이 CASCADE 처리; UNIQUE(post_id, user_id)]
users (1) ── (N) community_posts / community_comments / community_likes  [author_user_id, 앱이 CASCADE 처리]
```

## 1. 인증·사용자

### `users` — 사용자 (자체 인증, 소셜 전용)

원본 Supabase 인증을 대체하는 자체 사용자 테이블. 모든 도메인 테이블의 `user_id`가 참조한다. 카카오/구글/네이버 소셜 로그인 + 온보딩 완료 시점에 생성.

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | BIGINT | PK, IDENTITY | |
| `email` | VARCHAR(255) | NOT NULL, UNIQUE | 온보딩에서 항상 채워짐 (소셜 전용) |
| `nickname` | VARCHAR(100) | NOT NULL, UNIQUE | 계정 표시명/닉네임. **전역 유일**(`uq_users_nickname`, 정확 일치) — 향후 커뮤니티 기능 대비. 가게명과 분리(가게명은 `user_profiles.store_name`) |
| `provider` | VARCHAR(20) | NOT NULL, DEFAULT 'LOCAL' | 소셜 제공자 (KAKAO/GOOGLE/NAVER 등) |
| `provider_id` | VARCHAR(255) | | 소셜 측 사용자 식별자 |
| `is_active` | BOOLEAN | NOT NULL, DEFAULT TRUE | |
| `is_admin` | BOOLEAN | NOT NULL, DEFAULT FALSE | 운영 관리자 여부. 커뮤니티 공지(notice) 작성·비밀글/댓글 열람·타인 글 삭제 권한 판정 |
| `created_at` / `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 트리거로 updated_at 자동 갱신 |

> 과거 `onboarded` 컬럼은 제거됐다. User 행은 `register/complete`(온보딩 완료) 시점에만 `user_profiles`와 함께 원자적으로 생성되므로 **User 존재 = 온보딩 완료**이고, 별도 플래그가 불필요하다.

- **인덱스**: PK(id), UNIQUE(email), `uq_users_nickname` UNIQUE(nickname) — 닉네임 전역 유일, `uq_users_provider_identity` UNIQUE(provider, provider_id) WHERE provider_id IS NOT NULL — 같은 소셜 신원=같은 사용자
- **트리거**: `update_users_updated_at`
- **닉네임 유일성**: `nickname`은 NOT NULL이며 `uq_users_nickname` UNIQUE다. 가입(`register/complete`)·프로필 편집(`POST /me/profile`, 본인 제외 검사) 양쪽에서 중복을 거부하며, 동시성 경쟁 시 unique 위반을 잡아 `이미 사용 중인 닉네임입니다`(409)로 변환한다. 중복 충돌 메시지는 신원(`이미 가입된 계정입니다`)·이메일(`이미 사용 중인 이메일입니다`)·닉네임 셋으로 구분된다.

### `refresh_tokens` — 리프레시 토큰 저장소 (회전/세션 추적 + 통계)

JWT refresh 회전/무효화 추적. **원문이 아닌 SHA-256 해시만 저장**. 회전 시 이전 토큰을 `ROTATED`로, 로그아웃 시 `LOGGED_OUT`로 표시한다.
발급 컨텍스트(채널/기기/UA/IP)와 회전 계보(parent/세션시작/재발급수), 종료 사유(`status`)를 함께 축적해 추후 세션/로그인 통계에 활용한다.

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | BIGINT | PK, IDENTITY | |
| `user_id` | BIGINT | NOT NULL | 논리참조(FK 제약 없음) |
| `token_hash` | VARCHAR(64) | NOT NULL, UNIQUE | SHA-256 해시 (64 hex) |
| `parent_token_id` | BIGINT | NULL | 회전 계보: 직전(재발급원) 토큰 id |
| `client_id` | VARCHAR(64) | NULL | 채널 식별자(app/web). `X-Client-Id` 헤더 |
| `device_id` | VARCHAR(128) | NULL | 디바이스 식별자. `X-Device-Id` 헤더 |
| `user_agent` | VARCHAR(512) | NULL | 발급 시 User-Agent |
| `created_ip` | VARCHAR(45) | NULL | 발급 시 IP(IPv6 포함) |
| `status` | VARCHAR(20) | NOT NULL, DEFAULT 'ACTIVE', CHECK | `ACTIVE`/`ROTATED`/`LOGGED_OUT`/`EXPIRED` |
| `reissued_count` | INT | NOT NULL, DEFAULT 0 | 세션 내 재발급(회전) 횟수 |
| `session_started_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 세션 최초 시작(회전 시 계승) |
| `last_used_at` | TIMESTAMPTZ | NULL | 마지막 사용(회전/로그아웃) 시각 |
| `expires_at` | TIMESTAMPTZ | NOT NULL | 만료 시각 |
| `created_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | |
| `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 트리거 자동 갱신 |

- **인덱스**: `idx_refresh_tokens_user_id`, `idx_refresh_tokens_expires_at`, `idx_refresh_tokens_user_last_used (user_id, last_used_at)`, `idx_refresh_tokens_client (client_id)`
- 발급 컨텍스트는 `ClientContextFilter`가 요청 헤더/원격주소에서 캡처(`common/request/`). 필터를 거치지 않으면 NULL.

### `user_profiles` — 사용자 프로필 (가게 정보, users와 1:1)

온보딩에서 수집하는 가게 프로필. PK가 곧 `user_id`이므로 본질적으로 테넌트 격리(임의 user_id 주입 불가). FK 제약 없음 — 논리참조.

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `user_id` | BIGINT | **PK**→users 논리참조 | |
| `store_name` | TEXT | NOT NULL | 가게명 (`users.nickname`과 분리) |
| `region_sido` | TEXT | NOT NULL | 시/도 |
| `region_sigungu` | TEXT | | 시/군/구 |
| `owner_age_range` | TEXT | | 사장 연령대 |
| `interests` | TEXT[] | NOT NULL, DEFAULT '{}' | 관심사 (배열) |
| `specialties` | TEXT[] | NOT NULL, DEFAULT '{}' | 전문 분야 (배열) |
| `created_at` / `updated_at` | TIMESTAMPTZ | NOT NULL, DEFAULT NOW() | 트리거 갱신 |

### `user_preferences` — 사용자 설정 (하단바 커스터마이즈, users와 1:1)

| 컬럼 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `user_id` | BIGINT | **PK**→users 논리참조 | |
| `bottom_nav_items` | JSONB | NOT NULL, DEFAULT `["calendar","sales","expenses","customers","insights"]` | 하단 네비 항목 순서 |
| `updated_at` | TIMESTAMPTZ | DEFAULT NOW() | 트리거 갱신 |

## 4. 지출

### `expenses` — 지출

`total_amount`는 서버가 `unit_price * quantity`로 계산(SSOT). 카테고리 CHECK 없음 — 커스텀 카테고리 허용.

## 부록: 공통 규칙

- **`update_updated_at()` 트리거 함수**: `updated_at` 컬럼이 있는 거의 모든 테이블에 BEFORE UPDATE 트리거로 부착. 예외: `subscriptions`/`subscription_events`(앱 로직이 직접 갱신), `notification_log`/`recurring_skips`(updated_at 없음).
- **간접참조(FK 제약 없음)**: 모든 `user_id` 및 관계 컬럼은 DB FK 제약 없이 논리적 참조만 유지한다. 부모 행 삭제 시 연쇄 처리는 **애플리케이션 서비스가 책임**: 사용자 탈퇴 시 모든 사용자별 자식 행을 명시적으로 정리, `sales.customer_id` / `reservations.sale_id` / `photo_cards.sale_id` / `expenses.recurring_id`는 NULL로 설정(참조 보존 + 연결만 해제).
- **JSONB / 배열**: JPA에서 hypersistence-utils로 매핑. 통계 집계는 네이티브 SQL.
- **로컬 DB 초기화**: `docker compose down -v && docker compose up -d` 후 `docs/sql/all-tables-ddl.sql` + `docs/sql/seed.sql`을 수동 적용(DDL 직접 관리). 앱은 부팅 시 `validate`로 정합성만 검증.
